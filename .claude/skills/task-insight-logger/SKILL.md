# Task Insight Logger

## 目的

開発作業中に得た暗黙知を形式知として蓄積する。
タスクを遂行する過程で得た技術的知見・設計判断の根拠・今後の学習キーワードを
プロジェクトローカルのMarkdownファイルに追記し、個人のナレッジベースを構築する。

## 発動条件

以下のいずれかを検知したら、**ユーザーの指示を待たずに**この手順を実行する：

1. ユーザーが「完了」「できた」「次」「LGTM」「ありがとう」等、タスク終了を示す発話をした
2. 自分自身がタスクの最終ステップ（コミット、テスト通過、ファイル生成等）を完了した
3. ユーザーが明示的に別のタスクの話題を始めた（＝前タスクの暗黙的完了）

## 記録手順

### Step 1: インサイトの抽出

今完了したタスクを振り返り、以下の3カテゴリについて記録すべき内容を特定する。
**何も得るものがなかったタスクは存在しない。** 些細に見えても必ず何か1つは書け。

#### 1. 技術的知見 (Technical Findings)
- 使ったAPI・ライブラリ・ツールの挙動で発見したこと
- ハマったポイントとその解決策
- パフォーマンスや互換性に関する気づき
- 「こう書くとこうなる」という具体的な因果関係

#### 2. 設計判断の根拠 (Design Decisions)
- なぜこの構造・パターン・命名にしたのか
- 検討した代替案とその却下理由
- トレードオフの内容（何を得て何を捨てたか）
- 将来変更が必要になりそうな箇所とその条件

#### 3. 学習キーワード (Learning Keywords)
- タスク中に「もっと深く知りたい」と感じたトピック
- 関連する公式ドキュメント・RFC・仕様書の名前
- 検索に使うべき具体的なキーワード（英語推奨）
- 関連する概念・理論・パターン名

### Step 2: ファイルへの追記

以下の手順でファイルに書き込む：

```bash
# 日付を取得
DATE=$(date +%Y%m%d)
FILEPATH="docs/local/${DATE}.md"

# ディレクトリがなければ作成
mkdir -p docs/local
```

ファイルが存在しない場合は、先頭に以下のヘッダーを書く：

```markdown
# Daily Insights - YYYY/MM/DD
```

ファイルが既に存在する場合は、末尾に追記する。

### Step 3: 記録フォーマット

以下のフォーマットで追記する。**時刻は必ず含める。**

```markdown

---

## HH:MM - [タスクの簡潔な名前]

### Technical Findings
- [具体的な知見を箇条書き]

### Design Decisions
- **[判断内容]**: [根拠と代替案]

### Learning Keywords
- `keyword-en` - 日本語での補足説明
```

#### フォーマットルール

- タスク名は「何をしたか」が一目でわかる粒度にする（例: "HonoXのミドルウェアでCORS設定"）
- Technical Findingsは再現可能な具体性で書く。「〜だった」ではなく「〜すると〜になる」
- Design Decisionsは必ず「なぜ」を含める。結論だけ書くな
- Learning Keywordsは英語のキーワードを先に書き、日本語補足を添える
- 3カテゴリすべてに該当がない場合はそのセクションを省略してよい。ただし最低1カテゴリは書くこと

### Step 4: ユーザーへの報告

記録完了後、以下を簡潔に報告する：

```
📝 `docs/local/yyyymmdd.md` にインサイトを記録しました。
```

冗長な説明は不要。記録した内容の要約を1-2行で添えるだけでよい。

## .gitignore について

`docs/local/` はあくまで個人のローカルナレッジベースである。
初回実行時、`.gitignore` に `docs/local/` が含まれていなければ追加を**提案**する（強制はしない）。
チームで共有したいケースもあるため、判断はユーザーに委ねる。

## 注意事項

- このスキルはタスクの成果物とは独立して動く。タスクの成果物を汚染しない
- 記録は日本語で書く（Learning Keywordsの検索用語は英語）
- 既存ファイルの内容を上書き・削除しない。必ず追記のみ
- セッション中にファイルI/Oエラーが出た場合は、記録内容をチャット上に出力してフォールバックする
